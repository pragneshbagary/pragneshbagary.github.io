Replicating Soft-TAC: a reward-learning experiment on LunarLander

Reimplementing a 2026 RL paper's central claim from scratch - that a tanh-based loss degrades more gracefully than cross-entropy under label noise - and watching which results survived a second look.

I came across "Towards Improving Reward Design in RL" (Muslimani et al., 2025) and its follow-up, "The Trajectory Alignment Coefficient in Two Acts" (Muslimani et al., 2026). Both center on a metric called the Trajectory Alignment Coefficient (TAC), which measures how well a reward function matches what a human actually wants. My first read left one thing unclear: is TAC computed for an agent, or for a human? It's the latter. TAC is not a property of a policy's behavior. It is a diagnostic for a reward function, checking whether the trajectory rankings that reward function induces match a human stakeholder's own rankings. Once that distinction was clear, the rest of both papers followed more easily.

The second paper introduces Soft-TAC, a differentiable relaxation of TAC that can be used directly as a training loss. It comes with a specific claim: its tanh-based loss saturates on confidently-wrong preference pairs, so it should degrade more gracefully than a standard Cross-Entropy loss as label noise increases. Rather than take that claim at face value, I reimplemented the paper's LunarLander experiment from scratch to test it directly.

Building a synthetic ground truth

The original experiment requires a human to rate pairs of trajectories, which I don't have access to. Instead, I used a fixed, manually defined weight vector, w_star, and scored each trajectory as the dot product of w_star and an 8-dimensional feature vector, Phi (progress toward the pad, change in speed, tilt, leg contact, engine usage, crash, safe landing). Ranking trajectory pairs by that score produces preference labels in the same format a human labeler would produce.

This looked like a compromise at first, but it's closer to an advantage: it gives me a ground truth to check against. The original paper has no way to verify whether Soft-TAC recovered the true reward, since the authors never had access to their human labeler's internal weights, only their preferences. Because I constructed w_star myself, I can measure recovery directly, which is a check the original study could not perform.

I built the pipeline in stages and validated each one before moving to the next: the feature extractor first (checked against a handful of manual transitions), then the rollout that sums features into a per-episode Phi vector, then the synthetic labeling step, then the Soft-TAC loss in isolation, and only then the full training loop. Isolating each stage made it easier to localize problems later, rather than debugging the whole pipeline at once.

An early dataset that could not have worked

The first version of the pipeline generated trajectories from a random policy. Training ran without errors, but two of the eight features, crash and safe_land, had zero variance across the dataset: a random policy never lands a LunarLander, and it either times out or crashes.

This is a coverage problem, not a training problem. A feature with zero variance across the dataset gives a preference-learning method nothing to key off of, so its weight is unrecoverable regardless of the loss function used. The fix was a better dataset, not a better loss: I trained a DQN agent on LunarLander, saved checkpoints at several points during training (early checkpoints crash reliably while late ones land reliably), and rolled out episodes across that range of checkpoints so the final pool contained a real mix of crashes and landings.

A quiet bug in the feature extractor

At one point I found a bug in the feature extractor: the "progress" feature was computing the next distance to the pad using the current y-position instead of the next one, and the tilt-change feature was missing an absolute value (tilt direction shouldn't matter, only distance from upright). Neither error produced a visible failure. The pipeline ran end to end and the numbers looked plausible. They just measured something slightly different from what was intended.

Bugs like this are the ones worth taking most seriously, since everything downstream - labeling, training, and recovery - depends on the features being correct. An incorrect feature doesn't raise an error. It just silently invalidates every result built on top of it. Fixing it required regenerating the dataset and rerunning the full pipeline.

The noise sweep, and two results that didn't survive a second look

The core test of the paper's claim is a noise sweep: flip each preference label independently with probability p, retrain both losses from the same initialization, and measure how recovery (cosine similarity to w_star) degrades as p increases.

The first run of this sweep showed a small increase in Soft-TAC's recovery around p = 0.10 before it continued to decline, an effect that would have made for an appealing result on its own. Rather than report it, I reran the sweep with a disjoint set of seeds, and the effect did not reappear. It was seed variance, not a property of the loss. The same happened with an apparent crossover where Soft-TAC underperformed Cross-Entropy above p = 0.30: it also did not hold up under a different seed set. Both results would have been more interesting to report than what actually held up, but neither was real.

Feature scale was distorting the headline result

While preparing this post, I re-examined the noise-sweep numbers and found that recovery looked strong at p = 0 but Cross-Entropy in particular dropped sharply as soon as any noise was introduced (from about 0.98 to 0.23 between p = 0 and p = 0.05).

The cause was feature scale, not the loss function itself. Two of the eight features, side-engine and main-engine usage, are raw per-episode firing counts in the hundreds. The remaining features are order-1 or binary. Even though w_star assigns the engine features a small weight, their scale means they account for roughly 98% of the variance in the labeling score, versus under 1% each for crash and safe_land, which is the opposite of what w_star is meant to emphasize. Under label noise, gradient descent allocates more effective step size to whichever feature has the largest raw scale, so noise is absorbed almost entirely by the engine features rather than distributed across all of them. That is what produced the sharp collapse.

I addressed this by dividing each feature by its own standard deviation before labeling and training, without subtracting the mean. Every label and every loss in this pipeline depends only on a difference between two trajectories' features, so any constant subtracted from a feature cancels out of that difference. Centering would not have changed the result and would have distorted the 0/1 meaning of the binary features for no benefit. With this fix applied, the noise sweep changed substantially: both losses settled to roughly 0.80 recovery across nearly the entire noise range, and the separation between them largely disappeared.

This left a genuine tradeoff. The scaled version is the more principled setup, since it reflects what w_star's weights are meant to express, but it also removes the effect I set out to demonstrate. I chose to report the unscaled result as the primary finding, since it demonstrates the paper's central claim about tanh saturation more clearly, and added an explicit note that the separation is scale-dependent and does not survive standard feature normalization. I considered this preferable to reporting a more defensible setup that obscures the effect being tested.

Checking whether 0.81 cosine similarity means anything

At one point during debugging, I had a recovery value around 0.81 and no good sense of whether that number was meaningful. I checked it rather than guessing: cosine similarity between w_star and random 8-dimensional vectors averages around 0.29, so 0.81 is clearly not noise. I also checked whether additional training would improve it further. It did not move substantially, since the model was already classifying more than 99% of preference pairs correctly and had effectively solved the ranking task. Examining feature correlations explained the gap: several features are moderately to strongly correlated with each other (main-engine usage correlates at -0.86 with leg-contact state, for example), so more than one reward direction can explain nearly the same set of preferences. That is a data-coverage limitation rather than a training issue. More trajectories that break those correlations would likely help, whereas more training epochs or a different loss would not.

A reproducibility issue worth noting

Regenerating the dataset at one point produced a different number of landings and crashes than the committed dataset, despite using the same fixed random seed. DQN training in this setup is not bit-for-bit reproducible across runs. The committed dataset file is fixed, so the analysis built on top of it reproduces exactly. It is only regeneration from scratch that varies.

Open questions

  • Whether the unscaled result should be the one reported as primary, given that it does not survive standardization. I made a choice here, and I do not think it is the only reasonable one.
  • How stable a single training run really is without seed averaging. Running the clean-label case with no fixed seed 40 times gave a mean recovery of about 0.85 with a standard deviation of 0.15, and roughly one in five runs landed below 0.70. The loss is non-convex enough that initialization matters more than expected.
  • This is a replication of an existing claim, not new research. I believe it is an accurate one, but that distinction is worth stating plainly.

A divergence from the paper's setup

The paper's own LunarLander study uses the continuous-action variant of the environment. This implementation uses the discrete-action variant instead, since both the feature extractor and the DQN agent depend on the action being one of a fixed set of integers. The comparison between the two losses does not depend on this choice, since it only operates on the Phi feature vectors, but it is a real divergence from the paper's exact setup and worth stating directly.

From Scalar Alignment to Vector Fingerprints

One thing from the first paper has stuck with me: TAC collapses everything into a single number, one induced ranking of trajectories against one human ranking. I keep wondering whether that collapse is necessary. The reward here is eight-dimensional, but I only ever compare it to w_star as a whole, through cosine similarity or a per-feature weight table. What I haven't tried is running the ranking-agreement computation per dimension, trading the single alignment number for a vector of them, an "alignment fingerprint." The appeal is obvious: instead of "something is misaligned," you'd hope to see which objective.
I'm not sure it's that clean, though. The human ranking reflects all eight dimensions blended together, so a per-dimension agreement score measures how well one feature's ordering matches the full preference, which isn't the same as whether that feature is misaligned. A dimension could score low just because it only governs part of the preference. So the honest version of this idea has to work out what each per-dimension number actually means before it can claim to localize anything. My per-feature weight table gestures at the same intuition but isn't that computation.
I looked for this specific decomposition in the published work and didn't find it, though it's a small step from what's there and the field moves fast, so I'd call that "I didn't find it," not "it doesn't exist." Either way, it's where I'd pick this up next.
